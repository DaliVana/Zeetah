# **Comprehensive Vulnerability Analysis of Regular Expression Engines: A Threat-Informed Blueprint for Test Engineering**

The development of a modern regular expression engine requires navigating a complex intersection of formal language theory, finite automata algorithms, and low-level memory management. Regular expression engines are foundational components of contemporary software infrastructure, utilized pervasively for input validation, pattern matching, syntax highlighting, text processing, and deep data extraction.1 These engines are either integrated natively into language runtimes—such as Python’s re module and JavaScript's V8 Irregexp—or provided as highly optimized external libraries like PCRE2, RE2, and Oniguruma.1 Despite their ubiquity, the computational complexity of parsing logic and the intricate memory management required to process complex string patterns render these engines highly susceptible to critical security vulnerabilities.2

To engineer a robust test suite for a newly designed regular expression engine, one must systematically study historical failure modalities. Over the last decade, highly adopted engines have accumulated numerous Common Vulnerabilities and Exposures (CVEs). A comprehensive evaluation of these vulnerabilities reveals that they range from severe memory safety violations—which account for approximately 52% of all engine CVEs—to semantic anomalies and algorithmic complexity flaws.1 This report provides an exhaustive, low-level analysis of vulnerabilities across the most prominent regular expression engines. By dissecting the root causes of memory corruption, algorithmic denial of service, and compilation defects, this document establishes a foundational catalog of test cases. Furthermore, the analysis evaluates modern validation frameworks, specifically grammar-aware fuzzing and metamorphic testing derived from Kleene algebra, to establish a rigorous testing paradigm for next-generation engine development.

## **Theoretical Foundations and Architectural Paradigms**

To construct an adversarial test suite for a new regex engine, the vulnerability landscape must be understood through the lens of the underlying architectural paradigms that dictate pattern evaluation. Engines broadly fall into two distinct execution categories: Nondeterministic Finite Automata (NFA) driven by backtracking algorithms, and Deterministic Finite Automata (DFA) or Thompson NFA implementations that guarantee linear execution time.3 The architectural choice directly dictates the classes of vulnerabilities the engine will face, thereby informing the structure of the requisite testing matrix.

Backtracking engines, which include PCRE, Oniguruma, Python’s native re module, and Java's java.util.regex, construct a state machine that sequentially evaluates alternatives.3 When a particular evaluation path fails to produce a match, the engine backtracks to a previous decision point, discards the consumed input characters, and attempts the next alternative path. This design is highly flexible and enables advanced features that strictly exceed regular language constraints, such as backreferences (e.g., matching a previously captured group via \\1), look-around assertions (lookaheads and lookbehinds), and complex nested capture groups.3 However, this optimism introduces severe algorithmic vulnerabilities. When confronted with pathological input strings, the backtracking engine may be forced to explore an exponential number of failing paths before ultimately concluding that no match exists.3

Conversely, engines designed for security-critical environments, such as Google’s RE2, utilize a pessimistic, parallel evaluation model. RE2 constructs a finite automaton and evaluates all possible state alternatives simultaneously, advancing step-by-step through the input text.3 This architecture inherently immunizes the engine against catastrophic backtracking, mathematically guaranteeing ![][image1] execution time relative to the input string length.3 The trade-off is a deliberate omission of advanced features like backreferences, which inherently require non-linear memory and time complexity to resolve.3 Furthermore, dynamic DFA construction can lead to state space explosion; thus, engines like RE2 must enforce a strict, configurable memory budget, failing over to slower NFA evaluations if the cache is exhausted.3 Testing a new engine requires subjecting it to both algorithmic complexity stress tests and memory limit evaluations to ensure graceful degradation.

## **The Algorithmic Threat: Regular Expression Denial of Service (ReDoS)**

Regular Expression Denial of Service (ReDoS) is an algorithmic complexity attack specifically targeting backtracking NFA implementations.6 The attack transpires when an engine evaluates a vulnerable pattern against a carefully crafted input string, resulting in exponential (![][image2]) or high-polynomial execution time.7 Problematic patterns typically feature grouping with internal repetition, or alternations with overlapping capture scopes within a repeated group.8

The mechanics of ReDoS dictate specific testing methodologies. An adversarial test case typically consists of a vulnerable regex pattern, a prefix that initiates the vulnerable sub-pattern, a long sequence of characters that triggers ambiguous matches inside overlapping quantifiers, and a final terminal character that forces an overall failure, requiring the engine to evaluate every possible path.8 A canonical example is the pattern ^(a+)+$ evaluated against the string aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaX.5 The engine processes the contiguous block of identical characters efficiently but fails at the terminal character X. Because the nested quantifiers \+ offer exponentially overlapping ways to segment the string, the engine backtracks recursively. In server-side applications, a single maliciously crafted string can block the execution thread, exhaust CPU cycles, and render the application completely unresponsive.5

The National Vulnerability Database (NVD) contains over 400 ReDoS-specific CVEs disclosed since 2014, usually classified under CWE-1333 (Inefficient Regular Expression Complexity) or CWE-400 (Uncontrolled Resource Consumption).9 Any test suite for a new engine must implement a vast array of historical ReDoS payloads to verify the efficacy of the engine's backtracking limits, loop detection heuristics, or computational timeout implementations.

### **Prominent ReDoS Vulnerabilities for Test Case Generation**

Historical ReDoS vulnerabilities provide explicit patterns that must be incorporated into the test corpus. By recreating the exact conditions that crashed mature software systems, the new engine's resilience can be empirically validated.

**V8 Engine and Minimatch (CVE-2026-26996):** The V8 Irregexp engine, utilized by Node.js, suffered a critical ReDoS vulnerability via the minimatch library. The library compiled glob patterns into regular expressions, translating a wildcard \* into the non-greedy regex group \[^/\]\*?.11 When evaluating a glob pattern with many consecutive wildcards followed by a literal character (e.g., \*\*\*\*\*\*\*\*\*\*\*\*\*\*\*X\*\*\*), V8 compiled a regex containing multiple repeating, overlapping groups: /^(?\!\\.)\[^/\]\*?\[^/\]\*?...X\[^/\]\*?$/.12 If the target test string lacked the terminal character X, Irregexp backtracked through every possible permutation of character distribution across the ![][image3] consecutive groups. The time complexity scaled exponentially at ![][image4], meaning a pattern with ![][image5] wildcards would block the event loop indefinitely.12

**Ruby ActiveSupport (CVE-2026-33169):** The ActiveSupport component of Ruby on Rails utilized an inefficient regular expression for formatting numeric strings within the NumberToDelimitedConverter class.13 The regex exhibited quadratic time complexity when processing excessively long numeric strings, leading directly to CPU exhaustion.13 Testing a new engine requires evaluating deeply nested quantifiers applied to long contiguous numerical sequences to ensure linear time resolution.

**Python's re Module and Mistune (CVE-2026-33079):** The Python markdown parser mistune relied on the native re module to evaluate an overlapping regex pattern for inline link titles: (A|B)\* where both branches could match the sequence \\\!.14 Branch A matched \\\! as an escaped punctuation mark, while Branch B matched it as two separate literal characters \\ and \!.15 Providing a payload of repeating \\\! pairs with no closing quotation mark forced the Python re engine to explore an ![][image2] state space, blocking the parser entirely.15 This demonstrates that ambiguity within alternations is a primary driver of catastrophic backtracking.

**PraisonAI Python Framework (CVE-2026-34939):** This vulnerability highlights the danger of direct unvalidated compilation. The MCPToolIndex.search\_tools() function accepted user-supplied strings and compiled them directly via re.search() without any complexity guard or execution timeout.16 The execution blocked the Python Global Interpreter Lock (GIL) and the active thread for hundreds of seconds, achieving total service denial via network vectors.16

**Java java.util.regex Stack Exhaustion (CVE-2022-38751):** Java's native engine maps backtracking recursion directly to the hardware execution stack.8 The SnakeYAML parser processed untrusted YAML files using regex evaluation.19 Specially crafted inputs forced deep evaluation inside the java.util.regex.Pattern$Ques.match algorithm.20 The recursive matching depth exceeded the JVM stack limits, causing an uncaught StackOverflowError that crashed the host application.21 This vulnerability underscores a critical architectural requirement: a resilient regex engine must eschew recursive function calls mapped to the hardware stack in favor of heap-allocated, bound-checked state machines to prevent arbitrary depth crashes.18

## **Low-Level Memory Safety and Execution Vulnerabilities**

While algorithmic complexity attacks primarily degrade availability, memory safety violations in regex engines present catastrophic risks to system integrity and confidentiality. Engines implemented in languages without memory safety guarantees, such as C and C++ (e.g., PCRE2, Oniguruma, and the underlying logic of Python's re), are exceptionally vulnerable to buffer overflows, out-of-bounds access, and use-after-free conditions.1 A rigorous test suite must evaluate how the new engine handles bounds checking, buffer reallocation during AST construction, and pointer lifecycle management.

### **PCRE and PCRE2 Memory Violations**

The Perl Compatible Regular Expressions (PCRE) library and its successor PCRE2 are embedded in heavily utilized infrastructure, including Apache, Nginx, and PHP.24 Analyzing the specific memory mismanagement within PCRE2's source code provides highly specific unit testing criteria for array bounds and internal memory offsets.

**Heap-Buffer-Overflow via State Mismanagement (CVE-2025-58050):** This critical vulnerability in PCRE2 version 10.45 highlights the dangers of implementing complex control-flow directives within matching algorithms.23 The flaw occurs in src/pcre2\_match.c when combining the Scan SubString verb (\*scs:...) with the (\*ACCEPT) control verb.23 The (\*scs:...) verb isolates a portion of the pattern to evaluate against a specific captured substring, temporarily modifying the internal memory boundary pointers mb-\>end\_subject and mb-\>true\_end\_subject to restrict evaluation to the substring bounds.25

When the pattern inside the (\*scs:...) block successfully evaluates via an immediate (\*ACCEPT) directive, the engine forcibly terminates the current evaluation block. While the engine successfully restores the current subject instruction pointer (Feptr), it critically fails to restore the original values of the boundary pointers.25 These pointers remain stale, pointing to the end of the temporary substring instead of the actual subject boundary.25 Subsequent boundary checks calculate the remaining buffer length using these stale pointers. Due to unsigned integer casting (PCRE2\_SIZE), the miscalculation yields a massive positive number, bypassing bounds validations.25 A subsequent memcmp operation then reads data far beyond the actual allocated haystack buffer, resulting in severe information disclosure.25 Test engineers must design cases that utilize nested execution contexts with abrupt terminations to ensure the engine correctly unwinds and restores all parent state variables.

**Just-In-Time (JIT) Compilation Out-of-Bounds Reads (CVE-2022-1586, CVE-2022-1587):** To optimize execution, PCRE2 utilizes a JIT compiler to translate ASTs into machine code.26 CVE-2022-1586 manifested as an out-of-bounds read within the compile\_xclass\_matchingpath() function in pcre2\_jit\_compile.c.27 The issue centered on Unicode property matching (\\p{...}) during case-less execution. The JIT engine failed to fully read the character sequence length when parsing Property Character List (PT\_CLIST) operations without validating the preceding opcode type.27 Consequently, the emitted machine code accessed memory beyond allocated parameters.28 Similarly, CVE-2022-1587 involved an out-of-bounds read in get\_recurse\_data\_length() during the compilation of regexes containing duplicate data transfers in recursion loops.29 These flaws require the creation of test cases that heavily utilize Unicode property syntax combined with recursive references, specifically targeting engines that optimize ASTs into intermediate bytecode or native assembly.

**Integer Overflows in Quantifiers (CVE-2020-14155, CVE-2022-41409):** Integer overflow vulnerabilities frequently occur during the parsing of massive repetition quantifiers. PCRE CVE-2020-14155 allowed an integer overflow via an exceptionally large number following a (?C substring.29 CVE-2022-41409 demonstrated an integer overflow in pcre2test via negative input.29 The testing suite must include extreme boundary values for quantifiers (e.g., {2147483647}) to verify that the parsing logic securely traps overflow conditions rather than allocating undersized buffers.

### **Oniguruma Memory Corruption Vectors**

Oniguruma is an advanced regex library utilized primarily by the Ruby programming language, PHP's mbstring extension, and various Rust bindings.30 Its defining architectural trait is deep native support for multiple character encodings (e.g., UTF-8, UTF-16, EUC-JP, GB18030) directly within the pattern object.32 The intricate parsing required for multi-byte arrays generates unique memory corruption scenarios.

**Use-After-Free during Multi-byte Parsing (CVE-2019-13224):** This critical Use-After-Free (UAF) vulnerability resides in the onig\_new\_deluxe() function of regext.c in Oniguruma 6.9.2.30 The vulnerability occurs during the initialization and compilation of a regex pattern paired with specific multi-byte encoding definitions.30 When the engine encounters a crafted combination of pattern and encoding, it triggers an error state deep within the initialization sequence. The code frees the allocated regex structure but fails to nullify the reference pointer before returning control to a higher-level function. The engine subsequently attempts to access the freed memory block.30 By leveraging this UAF, an attacker supplying a precise payload can manipulate the heap layout to achieve arbitrary code execution or severe information disclosure.30 Test developers must ensure that the new engine's teardown sequences and exception handling paths are rigorously tested with invalid encoding sequences to verify pointer invalidation.

**Stack Out-of-Bounds Write via Case-Folding (CVE-2017-9225):** Unicode case-folding maps characters to normalized formats for case-insensitive matching. In Oniguruma 6.2.0, the onigenc\_unicode\_get\_case\_fold\_codes\_by\_str() function executed a stack out-of-bounds write.35 The issue stemmed from inadequate input validation of the maximum allowable Unicode code point. When presented with a malformed regular expression containing the invalid point 0xFFFFFFFF, the unicode\_unfold\_key() function failed to intercept the illegal value.35 The engine attempted to expand the case-folded string based on this maximum integer, causing expand\_case\_fold\_string() to write 4 bytes past the end of its allocated stack buffer.35 This classic stack buffer overflow allows an attacker to overwrite the return instruction pointer and redirect control flow.35 Test generation must focus on fuzzing invalid Unicode planes and boundary code points during case-insensitive evaluation phases.

**Heap-Based Over-reads and Integer Wraparound (CVE-2019-19012, CVE-2019-9023):** In CVE-2019-19012, the search\_in\_range function in regexec.c contained an unchecked integer arithmetic operation.38 When parsing exceedingly large quantifiers or offset parameters within a crafted pattern, the calculation wrapped around, producing a negative or undersized index. This bypassed subsequent boundary constraints, resulting in a heap-based buffer over-read.39 Furthermore, CVE-2019-9023 demonstrated widespread heap over-reads across regcomp.c, regexec.c, and regparse.c when the engine evaluated invalid multibyte sequences.40 A comprehensive test suite must intentionally inject malformed UTF-8 and UTF-16 byte sequences into patterns to validate that the parsing lexer immediately aborts execution rather than propagating corrupted offset indices into the evaluation phase.

## **V8 Irregexp Bytecode and JIT Vulnerabilities**

The Irregexp engine, native to Google's V8 JavaScript engine, represents the apex of regex optimization.1 Evaluating expressions involves a multi-stage pipeline: patterns are compiled into an internal bytecode, which is then interpreted or passed to the Sparkplug and TurboFan JIT compilers for aggressive native execution.26 The integration of a discrete virtual machine (the Irregexp VM) introduces highly specialized vulnerabilities that transcend standard C buffer overflows, manifesting as logic flaws within the bytecode compilation and execution phases.

Vulnerabilities within the Irregexp engine are critical due to their utilization in sandbox escape chains within web browsers. Modern exploits have demonstrated that manipulating the bytecode generated by Irregexp can yield arbitrary read and write primitives outside the V8 sandbox (e.g., V8 issue ID 344963941).44 Advanced Persistent Threat (APT) actors have historically leveraged these flaws; for instance, the Lazarus group utilized an Irregexp vulnerability (CVE-2024-4947) to bypass the V8 sandbox.43 The attack successfully modified the bytecode instructions to bypass boundary checks, allowing the JavaScript environment to read memory outside the bounds of the internal register array.43

Furthermore, the JIT compilation of regular expressions introduces the risk of type confusion. If the AST parser misidentifies a specific capture group or regex object prototype, the JIT compiler (TurboFan) may emit assembly instructions that improperly access the internal CloneObjectIC caches.44 Testing a new regex engine that features a bytecode or JIT layer requires adversarial inputs that specifically target the intermediate representation. Test methodologies must assert that modifications to the emitted bytecode cannot bypass the engine's internal bounds checks or corrupt the hardware execution context.

## **Polynomial Versus Exponential ReDoS: The Neglected Majority**

The canonical ReDoS narrative fixates on exponential blow-up (`^(a+)+$`), but the empirical record demonstrates that this framing is dangerously incomplete. The largest ecosystem-scale study of the problem—Davis, Coghlan, Servant, and Lee’s ESEC/FSE 2018 measurement of roughly 448,402 modules (66% of npm, 58% of PyPI)—found that approximately 1% of unique regexes are super-linear, present in 3% of npm modules and 1% of PyPI modules, affecting over 10,000 modules ecosystem-wide.49 Crucially, of the 3,589 super-linear regexes identified in npm, only 245 (7%) were exponential; 2,638 (74%) were quadratic, 535 (15%) cubic, and 144 (4%) were degree four or higher, with PyPI exhibiting an essentially identical distribution (6% exponential, 76% quadratic).49 The authors' own conclusion is unambiguous: a defense or test corpus that targets only exponential blow-up "will thus miss about 90% of SL regexes."49

This is not a theoretical concern. The most consequential real-world ReDoS incidents were polynomial, not exponential: the 2016 Stack Overflow outage was caused by quadratic backtracking on a trim regex, and the 2019 Cloudflare global outage was driven by a quartic (degree-4) pattern. A 2024/2025 systematization of knowledge confirmed that even after vendors deployed mitigations targeting exponential behavior, polynomial (quadratic and cubic) ReDoS continued to manifest in JavaScript and Java, and exponential behavior persisted in Ruby and Java.54 An engine claiming ReDoS resistance must therefore neutralize the entire polynomial hierarchy, not merely the exponential extreme.

The structural source of polynomial degree is precisely characterized. For a backtracking matcher, a pattern of the form `a*…a*` containing *k* overlapping quantified sub-expressions yields matching time that is polynomial of degree *k* on inputs of the form `aⁿx`: `a*a*` is quadratic, `a*a*a*` is cubic, and so on.52 Static detection maps this to two automata-theoretic ambiguity classes on the underlying NFA—**Exponential Degree of Ambiguity (EDA)**, which corresponds to exponential backtracking, and **Infinite Degree of Ambiguity (IDA)**, which corresponds to polynomial backtracking.52 Independent taxonomies converge on the same partition: ReDoSHunter (USENIX Security 2021) enumerates five root-cause patterns—Nested Quantifiers (NQ), Exponential Overlapping Disjunction (EOD), and Exponential Overlapping Adjacency (EOA) for the exponential class; Polynomial Overlapping Adjacency (POA) and Starting with a Large Quantifier (SLQ) for the polynomial class.50 RENGAR (IEEE S&P 2023) reduces every exponential vulnerability to a single quantified "pathological loop-subregex" pattern (EOLS) and splits polynomial vulnerabilities into two-loop (PTLS, e.g. `a+a+`) and single-loop (POLS, e.g. `a+$`) forms.51 These structural catalogs constitute a precise checklist that a test suite must exercise.

Zeetah's DFA-first architecture neutralizes the *backtracking* form of the degree hierarchy, which is its decisive advantage over mere "exponential-only" hardening: a single DFA *scan* of `a*a*`, `a*a*a*`, or arbitrarily many overlapping loops is O(n), and the EDA/IDA backtracking-ambiguity blow-up that felled Stack Overflow and Cloudflare cannot occur inside a single scan. The narrow residue—genuinely non-regular constructs routed to the tree backtracker—is bounded there by the explicit step budget rather than by automaton determinism.

### **The All-Matches Quadratic Trap**

A subtle corollary deserves explicit attention: even engines with a rigorous single-match linear-time guarantee (RE2, Go `regexp`, the Rust `regex` crate, .NET `NonBacktracking`) can degrade to O(n²) when an application enumerates *all* matches naively, because each successive search may re-scan from the previous match position over an input where matches are sparse.60 Zeetah's enumeration APIs are explicitly engineered against this trap: `findAll`, `iterator`, `capturesIterator`, and `count` are documented and implemented as O(n) overall rather than O(n · matches), resuming each successive search from the prior cursor rather than restarting the scan, and the `advanceEmpty` helper guarantees forward progress of at least one byte on empty matches (preventing both stalls and accidental re-scan amplification). The contract suite should assert that iterating all matches over a large input with many matches remains linear.

## **The Untrusted-Pattern Trust Boundary: Regex Injection**

The threat models above assume a *trusted pattern* matched against *untrusted input*. A distinct and frequently overlooked vulnerability class arises when the **pattern itself is attacker-controlled**—either fully, or by interpolating untrusted data into a pattern template. SEI CERT rule IDS08-J formalizes this: "An attacker may supply a malicious input that modifies the original regular expression in such a way that the regex fails to comply with the program's specification."47 The impact extends well beyond denial of service into logic subversion and data exfiltration, via three concrete vectors: **flag override** (untrusted input toggling matching options such as case-insensitivity or single-line mode), **greediness manipulation** (rewriting quantifiers to match more than intended and thereby expose adjacent sensitive data), and **grouping manipulation** (injecting parentheses and alternations to change the pattern's logical structure).47 A canonical exploitation injects `.*)|(.*` into a log-filter template such as `.*? +public\[\d+\] +.*<SEARCHTEXT>.*`, producing a pattern whose top-level alternation matches *any* line in the log, including private ones.47 Critically, escaping metacharacters (e.g. Java's `Pattern.quote()`) is **not** a complete mitigation; the verified guidance is that untrusted text should not be compiled into a pattern at all, or must be constrained to a strict whitelist, with neutralization treated only as defense-in-depth.47

A second, more insidious facet is **pattern disclosure across the client/server boundary**. Barlas, Du, Davis, and colleagues (ICSE 2022) demonstrated that web services routinely publish their input-sanitization regexes in HTML `pattern` attributes, client-side JavaScript validation, and—most revealingly—machine-readable API specifications (OpenAPI, RAML, API Blueprint).48 Under the "Consistent Sanitization Assumption" that client-visible patterns mirror server-side logic, a black-box attacker can read a vulnerable pattern directly off the published interface and craft a tailored ReDoS payload with no white-box access whatsoever. In their study of 1,000 HTML-form services and 475 API services, every confirmed exploitable case (6 services across 15 subdomains) was discovered through API specifications; none through HTML forms, because API specs leak substantially more sanitization detail.48 The lesson for engine and framework authors is that a published regex is an attack surface, and that the four preconditions for ReDoS—attacker-controlled input, a super-linear pattern, an unsafe engine, and insufficient mitigation—must be assumed to hold simultaneously unless one is structurally removed.48,54

Zeetah addresses this boundary at the engine level rather than the application level. Because pattern compilation is a typed, bounded operation, a hostile *pattern* cannot force unbounded work: every construction ceiling (NFA states, NFA edges, DFA states, HIR nodes, capture groups) raises `error.PatternTooComplex` at compile time, and any pattern routed to the backtracker executes under the per-haystack step budget, surfacing `error.MatchBudgetExceeded` rather than hanging. This means a service that *does* compile semi-trusted patterns (a search box, a user-defined filter rule) inherits a hard upper bound on both compile-time and match-time cost for free. What the engine cannot do—and what the documentation must state plainly—is prevent *semantic* regex injection (flag override, grouping subversion); that remains an application-layer obligation to whitelist or refuse untrusted pattern text. The contract suite exercises malformed, oversized, and adversarial *patterns* (not just inputs) to confirm they are rejected with a typed error rather than crashing or looping.

## **Anchoring Semantics as a Validation-Bypass Surface**

Anchor semantics are a concrete, frequently exploited validation-bypass vector. The `$` anchor is **permissive** in Python, PHP, Perl, PCRE, and .NET—it matches not only at end-of-string but also immediately before a trailing newline at the end of the subject, even without multiline mode—whereas it is **strict** in JavaScript.55 Consequently, a validator written as `^pattern$` over a permissive engine can be bypassed by appending a trailing `\n`: the payload `safe-value\nmalicious-payload` may pass a `^safe-value$` check while smuggling a second line past the validator into downstream processing (header injection, log injection, SSRF allowlist bypass). OpenSSF's hardening guidance is explicit: prefer the unambiguous absolute anchors `\A` and `\z` where supported, and where `$` is permissive, use an explicit deterministic form such as `\n?\z`.55 The hazard is compounded by per-platform marker divergence—Python exposes `\A` and `\Z` (not `\z`), JavaScript supports neither `\A` nor `\Z` nor `\z` and offers only `^`/`$`—so portable, audit-clean validators cannot assume a common anchor vocabulary.55

Zeetah follows the OpenSSF recommendation by construction. It supports the full absolute-anchor vocabulary—`\A` (absolute start), `\z` (absolute end-of-text), and `\Z` (end-of-text before an optional final newline)—alongside `^`, `$`, `\b`, and `\B`. Decisively, its `$` is **strict by default**: outside multiline mode, `$` matches only at the true end of the subject (`pos == input.len`), never before a trailing newline. The permissive "before-trailing-newline" semantics are available only through the explicitly named `\Z` token, and line-oriented `^`/`$` behavior must be opted into via the `(?m)` flag or the `multiline` compile option. This eliminates the silent newline-smuggling bypass at the engine level: a `^…$` validator in Zeetah cannot be defeated by a single appended newline the way it can in PCRE or Python. The security suite should assert that `$` rejects `value\n` and that `\A…\z` is offered as the recommended anchor pair for whole-input validation.

## **Unicode and Case-Folding Hazards**

Unicode handling is a rich source of both memory-safety bugs (already catalogued above for Oniguruma) and *semantic* validation bypasses. Three classes recur in the literature and practice: **case-folding collisions**, where characters such as the Kelvin sign (U+212A) fold to ASCII `k` or the Turkish dotless-ı disrupts naïve `i`/`I` folding, allowing a case-insensitive denylist to be evaded or a case-insensitive allowlist to over-match; **normalization bypasses**, where a validator operates on un-normalized text and an attacker supplies a different Unicode normalization form (NFC vs. NFD vs. NFKC) of a "forbidden" string that compares unequal byte-wise but renders or later-normalizes identically; and **`\w`/`\d`/`\b` cross-script divergence**, where the meaning of "word character," "digit," or "word boundary" silently shifts depending on whether the engine interprets these classes as ASCII or full Unicode, so that a pattern audited against ASCII assumptions behaves differently on non-Latin input. A correct hardening posture requires the engine to define a single, documented canonicalization order (normalize-before-match versus match-over-raw-bytes) and to make the ASCII-versus-Unicode semantics of character classes explicit rather than implicit.

Zeetah's response is deliberate scoping: it is a **byte-oriented** engine that performs **ASCII-only** case folding, applies **no Unicode normalization** to either pattern or input, restricts `\p{…}` property classes to the Latin-1 range, and rejects the combination of case-insensitivity with property classes (`(?i)\p{…}` → `error.Unsupported`) and the full-Unicode mode flag (`error.NotImplemented`) rather than half-implementing them. This is simultaneously a limitation and a security property. The limitation: case-insensitive matching and `\w`/`\d`/`\b` are ASCII-scoped, so a caller needing Unicode-aware case-insensitivity or normalization-stable validation must canonicalize input *before* handing it to the engine—a contract the documentation must state loudly to avoid the false sense of safety that triggers the collision and normalization bypasses above. The security property: by not performing Unicode case-folding or normalization expansion at all, Zeetah structurally avoids the entire class of case-fold buffer-overflow and integer-overflow defects exemplified by Oniguruma's CVE-2017-9225 (stack out-of-bounds write during case-fold expansion) and CVE-2019-19012—there is no case-fold expansion buffer to overflow and no maximum-code-point arithmetic to wrap. The test suite must therefore (a) confirm that malformed and oversized UTF-8 byte sequences in *input* never crash or over-read (matching is defined over bytes, not validated code points), and (b) document the ASCII-folding boundary as an explicit caller obligation.

## **Compile-Time Amplification and Backreference Complexity**

Two amplification vectors operate at compile time and at the non-regular fallback, respectively, and both are distinct from classic input-driven ReDoS.

**Counted-repetition expansion bombs.** Bounded quantifiers such as `a{1000}` or, worse, nested forms like `a{1000}{1000}` can inflate the compiled automaton's state count multiplicatively, exhausting memory or CPU during *compilation* before any input is ever matched. USENIX Security 2022's "Counting in Regexes Considered Harmful" (Turoňová et al.) demonstrated that this counting-driven blow-up can afflict even non-backtracking and counting-automata engines, which is why "linear-time matcher" is not by itself a complete defense.59 Zeetah bounds this vector with two independent ceilings: a per-quantifier limit `MAX_REPEAT = 1000` (counts beyond it are rejected at parse time with `error.Unsupported`), and a global `MAX_NODES = 4096` ceiling on the expanded HIR/NFA, so that a multiplicative bomb such as `a{1000}{1000}`—which would attempt to materialize on the order of 10⁶ nodes—is rejected with `error.PatternTooComplex` rather than allocated. The security suite asserts that oversized and nested counted repetitions, inverted `{m,n}` bounds, and quantifier-count integer overflows all fail cleanly at compile time.

**Backreference-induced super-linearity.** Backreferences are not a mere performance footnote; matching regexes with backreferences is **NP-hard in general**, and selective-memoization techniques that provably linearize ordinary super-linear "K-regexes" explicitly *do not* linearize backreference regexes—they only reduce the exponent.53 Recent work (2026) formalized a "Two-Phase Memory Finite Automaton" model showing backreferences can induce super-linear runtime even when the underlying NFA ambiguity is only linear, a scenario existing ReDoS detectors miss.58 The unavoidable engineering consequence is that any engine supporting backreferences must treat them as a budget-capped feature rather than a free one. Zeetah's design honors this: backreferences (together with look-around and atomic groups) are precisely the constructs that route away from the DFA path into the tree backtracker, where they execute under the explicit step budget and recursion-depth guard, returning `error.MatchBudgetExceeded` on a pathological input instead of hanging. This converts an NP-hard worst case into a bounded, typed failure—the only safe contract available for a feature that cannot be linearized.

## **The Production Mitigation Landscape (2023–2026)**

Surveying how mature engines actually defend against these classes clarifies which architectural choices eliminate which vulnerability families—and validates Zeetah's positioning. A mid-2025 systematization (ASIA CCS '25) classifies production engines by ReDoS safety as follows: the Rust `regex` crate and Go's `regexp` are **safe** by virtue of Thompson-style non-backtracking automata with linear-time guarantees; CPython's `re` is **not safe** (no defense); and JavaScript/V8, Ruby, C#/.NET, Java/OpenJDK, PHP, and Perl are only **partial**.54 The decisive practical caveat is that the strongest mitigations in the "partial" engines are **off by default**: V8's non-backtracking engine requires a Node.js feature flag,56 and .NET's `RegexOptions.NonBacktracking` must be explicitly requested,61 so the overwhelming majority of applications on those platforms receive no protection. Weaker mitigation strategies seen in the wild include search-state memoization (Ruby 3.2, 2022), resource and counter caps (PHP's PCRE `backtrack_limit`, Perl), and match-time timeouts (.NET since v4.5, 2012), but these reduce rather than eliminate super-linear matching and—in the timeout case—merely convert a hang into a thrown exception after the CPU has already been spent. Selective memoization (Davis, Servant, Lee, IEEE S&P 2021) is the strongest known retrofit for a backtracking core: it provably linearizes K-regex matching in input length and, with run-length-encoded memo tables, reaches constant memo space for 90% of super-linear regexes—but it explicitly excludes backreferences and warns that the memo table itself becomes a space-DoS surface that must be bounded.53

Zeetah's two-tier model places it in the *safe-by-default* camp alongside Rust and Go, with a critical distinction from V8 and .NET: there is no opt-in flag to forget. Every regular pattern compiles to a DFA walked in O(n) by default; the eager DFA falls back to a bit-identical lazy DFA (never to backtracking) when the state ceiling is reached; and only genuinely non-regular constructs reach the step-budgeted backtracker. This structurally removes ReDoS preconditions (2) "a super-linear pattern reaches an unsafe matcher" and (3) "an unsafe engine" for the entire regular subset, while bounding the irreducible non-regular residue with an explicit budget rather than an after-the-fact timeout. The space-DoS hazard that selective memoization introduces does not arise, because the DFA path uses bounded state tables and the backtracker is step-bounded rather than memo-bounded. | Engine | Core Architecture | ReDoS Status | Default-On? | Residual Vulnerability Class |

| Go `regexp` | Thompson NFA / DFA | Safe | Yes | All-matches O(n²) if enumerated naïvely |

## **Recent CVEs (2024–2026) for Test-Corpus Seeding**

Beyond the historical CVEs catalogued earlier, several recent disclosures extend the test corpus. **CVE-2024-27282 (Ruby)** is an arbitrary-memory-address *read* in the regex engine, reinforcing that even mature interpreted-language engines continue to surface memory-disclosure primitives through their regex subsystem.57 **path-to-regexp** (the Express.js route-pattern compiler) was found ReDoS-vulnerable when route definitions with adjacent parameters compile to a backtracking-prone pattern, a reminder that *pattern-generating* libraries inherit the ReDoS surface of their target engine.62 The 2026 "ReDoS Induced by Backreferences" work (arXiv 2602.21459) introduced the Two-Phase Memory Finite Automaton model and identified novel backreference vulnerability patterns invisible to prior detectors.58 Each should be reproduced as a regression entry: the Ruby case validates memory-safe rejection of malformed patterns, path-to-regexp validates that route-style adjacent-parameter patterns compile to a DFA or are budget-bounded, and the backreference cases validate the step-budget contract on the non-regular path.

## **Advanced Testing Methodologies and the ReTest Framework**

The historical record dictates that traditional testing methodologies—namely regression testing and naive fuzzing—are grossly insufficient for discovering deep logic and memory flaws in regular expression engines.1 The reliance on differential testing, where the output of the engine is compared against reference implementations (e.g., comparing a new engine's output to PCRE), is fundamentally flawed due to the lack of a universal regex standard.1 Engines implement distinct dialects with subtle semantic differences, causing differential testing to generate massive volumes of false positives that mask genuine defects.1

A comprehensive academic framework, defined as *ReTest* (Towards the Systematic Testing of Regular Expression Engines), outlines a paradigm shift necessary for modern engine validation.1 The framework mandates the synthesis of two complementary techniques: grammar-aware fuzzing and metamorphic testing derived from Kleene algebra.1

### **Grammar-Aware Fuzzing and Agentic Strategies**

Traditional byte-level fuzzers (such as LibFuzzer) operate via random mutation.1 When applied to regex engines, these mutations almost exclusively generate syntactically invalid strings that fail at the primary parsing phase, exercising only the outermost error-handling logic and failing to penetrate the core matching internals or JIT compilation paths.1

To achieve maximal code and edge coverage, the test suite must implement grammar-aware fuzzing.1 This methodology restricts mutations to syntactically valid constructs defined by the engine's specific AST grammar. By ensuring the generated patterns are valid, the fuzzer forces the engine to compile the pattern, generate intermediate bytecode, and execute the matching algorithm, thereby probing deep memory bounds issues and catastrophic algorithmic complexity limits.1

Advanced implementations, such as the AFuzz (Agentic Fuzzing) protocol utilized on the V8 engine, leverage Large Language Models (LLMs) to construct highly targeted scenarios.45 By feeding historical root causes (such as the mb-\>end\_subject failure from PCRE CVE-2025-58050) into the agent, the fuzzer generates diverse PoC patterns designed to trigger similar state mismanagement across completely different code structures.45 The test suite for the new engine should programmatically generate thousands of variations of historical CVE patterns using an AST-aware generator.

### **Metamorphic Testing via Kleene Algebra**

To solve the test oracle problem—determining if a successful execution actually produced the semantically correct result without relying on flawed differential testing—engineers must employ Metamorphic Testing.1 Metamorphic testing validates that fundamental mathematical properties hold true within a single engine's execution, eliminating false positives.1

The *ReTest* framework implements 16 implementation-independent metamorphic relations derived from Kleene algebra.1 Because regular expressions are fundamentally algebraic structures, any formally correct regex engine, regardless of dialect, must satisfy these relations.1 During automated testing, the framework generates a highly complex, valid regular expression ![][image6] and evaluates it against an input string, capturing the result. It then evaluates the string against a mathematically equivalent pattern based on a specific relation. If the engine yields different match results, a semantic logic bug or internal optimization failure is immediately confirmed.1

The integration of these 16 metamorphic relations alongside grammar-aware fuzzing has been empirically proven to increase edge coverage by ![][image7] compared to standard techniques, successfully identifying new memory safety defects in mature engines like PCRE.1 The following table defines critical Kleene algebra metamorphic relations that must be hardcoded into the new engine's test oracle.

| Metamorphic Relation Name | Algebraic Representation | Testing Objective within the Regex Engine |
| :---- | :---- | :---- |
| Idempotence of Closure | ![][image8] | Verifies that nested un-bounded quantifiers are correctly optimized into a single evaluation path without triggering algorithmic recursion bugs. |
| Union Commutativity | $a | b \\equiv b |
| Idempotence of Union | $r | r \\equiv r$ |
| Associativity of Concatenation | ![][image9] | Evaluates the integrity of the state machine's state transitions when combining sequential matching blocks. |
| Distributivity of Concatenation over Union | $a(b | c) \\equiv ab |

## **Strategic Imperatives for Engine Architecture and Test Design**

The synthesis of historical CVE analysis and advanced validation methodologies dictates a clear blueprint for engineering and testing a new regular expression engine. The data demonstrates that permissive architectural choices inevitably culminate in catastrophic security failures.

First, the profound prevalence of memory corruption vulnerabilities—such as the heap buffer over-reads in PCRE2 and the out-of-bounds stack writes in Oniguruma—demands rigid memory governance.23 If the engine is constructed in C or C++, all state transitions, particularly those involving advanced control verbs like (\*ACCEPT) or look-arounds, must utilize atomic transactional macros that verify pointer integrity during context switches.25 The test suite must heavily fuzz the AST teardown sequences, garbage collection algorithms, and Unicode plane bounds checking.35 Zeetah sidesteps this entire vulnerability class at the language level: it is a pure-Zig library with zero external dependencies, so allocations are bounds-checked and lifecycle-managed by the host allocator, and the recursion-depth guard means even the backtracking tier does not map matching depth onto an unbounded hardware stack. The C-level over-reads, use-after-frees, and stack-overwrite primitives catalogued above have no direct analogue in a memory-safe Zig surface, but the test suite still exercises malformed and oversized patterns to confirm graceful, typed rejection.

Second, to mitigate the omnipresent threat of Regular Expression Denial of Service (ReDoS), the new engine should ideally default to a Thompson NFA or DFA architecture, similar to RE2.3 This paradigm mathematically guarantees linear execution time *per scan* relative to the input string, neutralizing the *exponential* ![][image2] backtracking blow-up inherent to backtracking engines. It does not by itself guarantee linear *search*: the quadratic and cubic cases that constitute roughly 90% of real-world super-linear regexes and caused the Stack Overflow and Cloudflare outages can still arise at the search layer (e.g. unanchored `$`-anchored patterns that restart the scan from every offset), which is exactly why a DFA-first engine must also implement single-pass unanchored leftmost search—which Zeetah now does for this class via a single-pass reverse-reachability search (its earlier O(n²) limitation in that path was surfaced by a deliberately failing test rather than papered over, and is now fixed).3,49 If product requirements mandate the inclusion of non-regular features like backreferences, the engine must implement a hybrid execution model: executing as a DFA by default and only falling back to a computationally bounded, timeout-enforced backtracking virtual machine when strictly necessary.5

This is precisely the model Zeetah adopts. Every pattern that lies within the regular subset is compiled to a deterministic finite automaton and walked as a table, yielding O(n) time *per scan* by construction; classic "catastrophic" regular shapes such as `(a+)+$`, `(a*)*$`, and `(.*)*$` compile (rather than being rejected) and—crucially—never exhibit the *exponential* backtracking blow-up of a PCRE-style engine. They are now linear end-to-end for this class too: as documented in the "Resolved" callout above, an unanchored `$`-anchored pattern with no required byte (including `(a+)+$` and `(a*)*$` on non-matching input, and the minimal `a+$`) used to incur an **O(n²)** leftmost search because the scan restarted from every offset, but is now routed to a single-pass reverse-reachability search (`lazy_dfa.findAnchoredEndFrom` / `DenseSearch.findFrom` with `a_end`) that is O(n)—`tests/security.zig` keeps a permanent regression guard for it. When the eager DFA would exceed its state ceiling, Zeetah falls back to a bit-identical lazy DFA — never to backtracking. Only genuinely non-regular constructs (backreferences, look-around, atomic groups, and lazy quantifiers combined with an end-anchor) are routed to a tree backtracker, and that backtracker executes under an explicit step budget (`8000 + (len+1)*4000`) plus a recursion-depth guard (`MAX_DEPTH = 16384`). Exceeding either limit returns a typed `error.MatchBudgetExceeded` at match time rather than hanging the thread — the .NET-style runtime contract. Construction ceilings (NFA states, NFA edges, DFA states, HIR nodes, and capture groups) likewise raise `error.PatternTooComplex` at compile time for patterns too large to build, so an adversary cannot force unbounded compilation either.

Finally, the testing infrastructure must abandon ad-hoc methodologies. The test suite must be built upon a foundation of grammar-aware fuzzing that is seeded by the structural patterns of historical CVEs, combined with a continuous evaluation of the 16 Kleene algebra metamorphic relations.1 By enforcing algebraic consistency at runtime and subjecting the engine to the precise mathematical edge cases that dismantled its predecessors, developers can ensure the delivery of a resilient and secure pattern-matching architecture. In Zeetah, `zig build test` runs the internal unit suite alongside dedicated feature and contract files, including a `tests/security.zig` suite that asserts the step-budget and construction-ceiling contracts above hold against the adversarial shapes catalogued in this report.
